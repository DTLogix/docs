# Персони користувачів DTLogix

## Вступ

DTLogix — цифрова платформа для транспортно-експедиційних компаній, яка централізує управління рейсами, ETA та сповіщеннями клієнтів і економить експедитору 1–2 години щодня за рахунок автоматизації.

Персони базуються на реальних болях з Problems та сегментах з Customer Segments. Вони допомагають приймати продуктові рішення та формувати маркетингові повідомлення.

---

## Персона 1: Соло-експедитор

### Базові дані

- **Ім'я:** Олексій
- **Вік:** 32 роки
- **Посада:** Соло-експедитор
- **Розмір команди:** 1 особа
- **Кількість рейсів/місяць:** 30–40
- **Сегмент:** A - Соло-експедитори / малі команди

### Контекст роботи

Олексій працює сам, веде всі рейси в Excel і через Viber. Сам відповідає клієнтам, сам шукає перевізників, постійно перемикається між чатами та таблицями. Щодня отримує десятки дзвінків і повідомлень від клієнтів із запитанням "де вантаж" та "коли буде доставка".

### Болі та проблеми

Детальний опис проблем клієнтів: [/business-model/1-problems.md](/business-model/1-problems.md).

**Ключові болі для Олексія (специфічні для соло-експедитора):**
- Відсутність єдиного місця для управління рейсами (тримає все "в голові" та в Excel)
- Постійні дзвінки від клієнтів (10-15 разів на день з запитанням "де вантаж")
- Хаотичне зберігання документів та фото (втрата інформації)

### Потреби та цілі

- Економія часу (1–2 години щодня) — головний тригер покупки.
- Потреба у швидкому впровадженні та простому інтерфейсі.
- Автоматичні сповіщення клієнтам для зменшення дзвінків.
- Клієнтський лінк, щоб клієнт міг сам переглядати статус.
- Централізоване зберігання документів та фото.

### Поведінка

- Швидко приймає рішення, бо немає часу на довгі роздуми.
- Готовий спробувати новий інструмент, якщо він одразу економить час.
- Найкоротший цикл від першого контакту до оплати (до 7 днів), якщо продукт одразу закриває проблему "хаосу".
- Простота онбордингу — ключ до швидкого adoption.

### Цінність DTLogix для Олексія

- Економія 1–2 години щодня через автоматизацію.
- Зменшення кількості дзвінків від клієнтів на 50–60%.
- Всі рейси в одному місці, без плутанини.
- Швидкий онбординг (10–15 хв) без складного навчання.

## Персона 2: Керівник малої логістичної компанії

### Базові дані

- **Ім'я:** Марія
- **Вік:** 38 років
- **Посада:** Керівник відділу логістики
- **Розмір команди:** 4–5 осіб
- **Кількість рейсів/місяць:** 70–100
- **Сегмент:** B - Малі логістичні компанії

### Контекст роботи

Марія керує командою з 4–5 осіб. Команда веде 70–100 рейсів на місяць. Інформація розкидана між різними менеджерами, часто виникають помилки через багатозадачність. Важливо мати контроль над SLA та історією рейсів.

### Болі та проблеми

Детальний опис проблем та сегментів: [/business-model/1-problems.md](/business-model/1-problems.md), [/business-model/2-customers-segments.md](/business-model/2-customers-segments.md).

**Ключові болі для Марії (специфічні для керівника малої команди):**
- Хаос у комунікаціях між членами команди
- Неможливість швидко передати рейс іншому менеджеру
- Відсутність історії рейсів для контролю SLA
- Часті помилки через багатозадачність

### Потреби та цілі

- Контроль над всіма рейсами команди.
- Швидка передача рейсів між менеджерами.
- Історія змін статусів та взаємодій.
- Контроль SLA та аналітика для керівників.
- Зменшення помилок через автоматизацію.

### Поведінка

- Приймає рішення після демонстрації та аналізу.
- Потребує кастомних onboarding-скриптів для команди.
- Готова платити більше за систему, яка зменшує помилки.
- Важливіша якість та контроль, ніж простота.
- Сегмент B забезпечить більший LTV через регулярний об'єм рейсів.

### Цінність DTLogix для Марії

- Централізоване управління всіма рейсами команди.
- Можливість швидко передати рейс іншому менеджеру.
- Історія рейсів для аналізу та контролю SLA.
- Зменшення помилок через автоматизацію статусів.
- Аналітика для прийняття рішень.

## Персона 3: Клієнт вантажовласник

### Базові дані

- **Ім'я:** Андрій
- **Вік:** 45 років
- **Посада:** Власник бізнесу / Логіст
- **Розмір команди:** [не застосовується]
- **Кількість рейсів/місяць:** 5–10 (як клієнт)
- **Сегмент:** C - Клієнти експедиторів

### Контекст роботи

Андрій працює з експедиторами для перевезення вантажів. Йому потрібно знати, де знаходиться вантаж та коли він буде доставлений. Зараз він постійно дзвонить експедитору, щоб дізнатися статус, що створює стрес та недовіру.

### Болі та проблеми

Детальний опис сегменту C та проблем: [/business-model/2-customers-segments.md](/business-model/2-customers-segments.md#сегмент-c).

**Ключові болі для Андрія (клієнт експедитора):**
- Відсутність прозорості (не розуміє, чому рейс затримується)
- Постійна необхідність дзвонити експедитору для отримання статусу
- Відсутність прогнозованого ETA для планування

### Потреби та цілі

- Очікують прозорості та прогнозованого ETA.
- Хочуть отримувати сповіщення без дзвінків.
- Можливість самостійно переглядати статус рейсу в реальному часі.
- Автоматичні сповіщення про зміну статусу та ETA.
- Довіра до експедитора через прозорість.

### Поведінка

- Не реєструється в системі експедитора, використовує клієнтський лінк.
- Перевіряє статус рейсу кілька разів на день.
- Очікує автоматичних сповіщень про важливі зміни.
- Прозорість впливає на вибір експедитора.
- Критично важливі сповіщення та ETA → вони впливають на adoption експедиторами.

### Цінність DTLogix для Андрія

- Прозорість: бачить статус рейсу та ETA в реальному часі.
- Автоматичні сповіщення без необхідності дзвонити.
- Довіра до експедитора через прозорість.
- Зменшення стресу та хвилювань.
- Можливість планувати свої процеси на основі точного ETA.

---

## Взаємозв'язок персон

- **Олексій (експедитор)** працює з **Андрієм (клієнтом)**.
- **Марія (керівник)** керує командою, яка працює з клієнтами типу **Андрія**.
- **Андрій (клієнт)** впливає на adoption DTLogix через вимоги прозорості.
- Прозорість для **Андрія** зменшує навантаження на **Олексія** та **Марію**.

## Гіпотези про персони

- Найшвидше конвертується сегмент A (Олексій), бо проблема найбільш відчутна.
- Сегмент B (Марія) забезпечить більший LTV через регулярний об'єм рейсів.
- Клієнтам експедиторів (Андрій) критично важливі сповіщення та ETA → вони впливають на adoption.
- Сегмент A має найкоротший цикл від першого контакту до оплати (до 7 днів).

## Наступні дії / наслідки

- Використовувати персони для формування маркетингових повідомлень.
- Пріоритизувати функції на основі потреб персон.
- Тестувати UI/UX з реальними користувачами, які відповідають персонам.
- Валідувати гіпотези про поведінку персон через інтерв'ю.
