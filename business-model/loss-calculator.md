# Калькулятор втрат (Lead Magnet)

> **Lead Magnet** — це корисна пропозиція або безкоштовний матеріал, який компанія або фахівець надає потенційному клієнту в обмін на його контактні дані (зазвичай email або номер телефону).

## Мета

Інтерактивний калькулятор для лендингу DTLogix, який:

- показує логістам їхні втрати часу, грошей і клієнтів через ручні процеси;
- формує "аутичний ефект" усвідомлення проблеми;
- збирає контактні дані після розрахунку;
- служить валідаційним інструментом для тестування попиту на DTLogix.

**Контекст:** Детальніше про використання калькулятора як каналу залучення: [/business-model/5-channels.md](/business-model/5-channels.md#consideration). Калькулятор як частина нечесної переваги: [/business-model/9-unfair-advantage.md](/business-model/9-unfair-advantage.md).

## User Stories

- **U1**: Як експедитор, я ввожу параметри свого процесу та отримую оцінку втрат у грошах/місяць.
- **U2**: Я бачу, як ці втрати можуть скоротитися після впровадження DTLogix (перемикач "До/Після").
- **U3**: Я залишаю контакт, щоб отримати персоналізований звіт і можливість отримати ранній доступ до майбутньої інформаційної системи.
- **U4**: Адмін/маркетинг отримують агрегацію відповідей у Базі даних / CRM для валідації.

## UX-потік користувача

- **Intro** - короткий текст і кнопка «Розрахувати втрати».
- **Калькулятор** - форма з 15+ параметрами.
- **Результат** - блок "До / Після DTLogix" + пояснення + графік/індикатор.
- **CTA** - форма збору контакту (email/телефон)
- **Підтвердження** - короткий успішний стан + посилання на отримання раннього доступу до майбутньої інформаційної системи.

## Поля калькулятора з поясненнями

### 1. Базові параметри роботи

|  Поле  |  ID  | Default | Пояснення |
| ------ | ---- | ------- | --------- |
| Кількість активних рейсів одночасно | `trips_concurrent` | 10 | Скільки рейсів у тебе під контролем зараз або в середньому кожного дня. Це визначає масштаб навантаження. |
| Час на ручний контроль одного рейсу (хв/день) | `ctl_minutes_per_trip` | 10 | Скільки хвилин у день ти витрачаєш, щоб перевірити статус одного рейсу — дзвінки, чати, таблиці. |
| Вартість години твого часу (грн) | `hourly_rate` | 500 | Орієнтовна ринкова або власна ставка за годину управлінської роботи (наприклад, 500 грн). Використовується для розрахунку фінансових втрат. |
| Робочих днів у місяці | `workdays_per_month` | 22 | Зазвичай 20–26. Множиться на щоденні витрати для отримання місячних значень. |

### 2. Комунікації з клієнтами та водіями

|  Поле  |  ID  | Default | Пояснення |
| ------ | ---- | ------- | --------- |
| Кількість запитів "де вантаж?" від клієнта на рейс/день | `client_pings_per_trip` | 3 | Скільки разів клієнт у середньому дзвонить або пише, щоб уточнити статус вантажу. |
| Скільки хвилин займає відповідь на один запит клієнта | `minutes_per_client_ping` | 5 | Включає пошук інформації, уточнення у диспетчера, написання відповіді. |
| % випадків, коли водій не виходить на зв'язок | `driver_nonresponse_rate` | 20 | Частка ситуацій, коли водій не відповідає або затримує відповідь, створюючи інформаційний вакуум. |
| Час на відновлення інформації (хв) | `minutes_to_recover_info` | 15 | Скільки часу займає пошук фактичної інформації у разі "мовчання" водія. |

### 3. Помилки, штрафи, простої

|  Поле  |  ID  | Default | Пояснення |
| ------ | ---- | ------- | --------- |
| % рейсів з помилками у даних (адреса, час, статус) | `data_error_rate` | 10 | Як часто трапляються неточності у таблицях чи звітах, що призводять до плутанини або штрафів. |
| Середній штраф або компенсація клієнту через помилку (грн) | `cost_per_data_error` | 2000 | Грошова вартість наслідків однієї помилки — наприклад, знижка клієнту, штраф чи втрата маржі. |
| % рейсів з простоєм або демереджем | `demurrage_trips_share` | 5 | Частка випадків, коли автомобіль затримався через відсутність координації, неправильний ETA тощо. |
| Середня вартість простою/демереджу (грн) | `cost_per_demurrage` | 2500 | Орієнтовні фінансові втрати за один простій. |

### 4. Втрата клієнтів і довіри

|  Поле  |  ID  | Default | Пояснення |
| ------ | ---- | ------- | --------- |
| Кількість активних клієнтів на місяць | `monthly_clients` | 10 | Скільки клієнтів замовляють перевезення в середньому щомісяця. |
| % клієнтів, які йдуть через операційні проблеми | `churn_rate_due_to_ops` | 2 | Відтік через запізнення, помилки, відсутність інформації — показник втрати лояльності. |
| Середня цінність одного клієнта (CLV, грн) | `clv` | 50000 | Скільки прибутку приносить клієнт за період співпраці. Дає змогу оцінити грошовий ефект відтоку. |

### 5. Внутрішня операційна рутина

|  Поле  |  ID  | Default | Пояснення |
| ------ | ---- | ------- | --------- |
| Час на підготовку заявки / PDF / PO (хв/день) | `minutes_pdf_order_prep` | 20 | Скільки хвилин займає створення документів вручну або копіювання шаблонів. |
| Час на складання зведеного статус-звіту (хв/день) | `minutes_status_report` | 30 | Скільки часу витрачаєш на оновлення статусів у таблиці або підготовку звітів клієнтам. |
| Час на пошук потрібних документів (хв/день) | `minutes_search_docs` | 20 | Витрачений час на пошук даних у чатах, таблицях або пошті. |

## Формули розрахунку

### 1. Втрати часу (год/день)

```ini
time_control_per_day = (trips_concurrent * ctl_minutes_per_trip) / 60
time_client_comms_per_day = (trips_concurrent * client_pings_per_trip * minutes_per_client_ping) / 60
time_driver_recovery_per_day = (trips_concurrent * (driver_nonresponse_rate/100) * minutes_to_recover_info) / 60
time_internal_ops_per_day = (minutes_pdf_order_prep + minutes_status_report + minutes_search_docs) / 60

time_total_per_day = time_control_per_day
                   + time_client_comms_per_day
                   + time_driver_recovery_per_day
                   + time_internal_ops_per_day
```

### 2. Втрати часу (год/місяць)

```ini
time_total_per_month = time_total_per_day * workdays_per_month
```

### 3. Прямі грошові втрати від часу (грн/міс)

```ini
cost_time_loss_per_month = time_total_per_month * hourly_rate
```

### 4. Грошові втрати від помилок/демереджу (грн/міс)

```ini
errors_per_month = trips_concurrent * workdays_per_month * (data_error_rate/100)
cost_errors_per_month = errors_per_month * cost_per_data_error

demurrage_events_per_month = trips_concurrent * workdays_per_month * (demurrage_trips_share/100)
cost_demurrage_per_month = demurrage_events_per_month * cost_per_demurrage
```

### 5. Втрати від відтоку клієнтів (грн/міс, усереднено)

```ini
churn_clients_per_month = monthly_clients * (churn_rate_due_to_ops/100)
cost_churn_per_month = churn_clients_per_month * clv
```

### 6. Підсумок "До DTLogix"

```ini
loss_total_before = cost_time_loss_per_month
                  + cost_errors_per_month
                  + cost_demurrage_per_month
                  + cost_churn_per_month
```

## "Після DTLogix" — моделювання економії

### 1. Коефіцієнти ефективності "Після DTLogix"

|  Поле  |  ID  | Default | Пояснення |
| ------ | ---- | ------- | --------- |
| Скорочення часу ручної роботи (%) | `k_time` | 60 | Очікувана частка автоматизації рутинних процесів (наприклад, -60%). |
| Зменшення кількості помилок (%) | `k_errors` | 50 | Скільки неточностей у даних можна усунути завдяки централізованому контролю. |
| Скорочення простоїв/демереджу (%) | `k_demurrage` | 30 | Як автоматичні ETA і трекінг допомагають уникати затримок. |
| Зменшення відтоку клієнтів (%) | `k_churn` | 20 | Як підвищення прозорості знижує втрати клієнтів. |

### 2. Розрахунок "Після"

```ini
cost_time_loss_after = cost_time_loss_per_month * (1 - k_time/100)
cost_errors_after    = cost_errors_per_month   * (1 - k_errors/100)
cost_demurrage_after = cost_demurrage_per_month* (1 - k_demurrage/100)
cost_churn_after     = cost_churn_per_month    * (1 - k_churn/100)

loss_total_after = cost_time_loss_after
                 + cost_errors_after
                 + cost_demurrage_after
                 + cost_churn_after

savings_abs = loss_total_before - loss_total_after
savings_pct = (savings_abs / loss_total_before) * 100  // якщо loss_total_before > 0
```

Додатково показати економію годин/міс:

```ini
saved_hours = time_total_per_month * (k_time/100)
```

## Результати та візуалізація

### 1. Огляд UX

Після натискання **«Порахувати»** користувач бачить:

- **KPI-картки** (верхній ряд) — головні метрики "До"/"Після".
- **Пояснювальний блок** — коротка інтерпретація результатів простими словами.
- **Візуалізації**:
  - порівняльна **колонкова діаграма** "До/Після" (загальна сума втрат);
  - **склад втрат до** (stacked bar або простий список з відсотками);
  - **зекономлені години/міс** (гейдж/індикатор або числова картка).
- **CTA-блок** — «Отримати персональний звіт + демо», «Подивитися можливості DTLogix».

> **Примітка:** Усі візуалізації — без зовнішніх бібліотек; доступні альтернативні "плоскі" варіанти (лише цифри), якщо дизайн секції мінімалістичний.

### 2. Поля результатів (ідентифікатори та опис)

#### 2.1. Основні KPI (перший ряд)

|  Поле  |  ID  |  Опис  |
| ------ | ---- | ------ |
| Втрати до (грн/міс) | `loss_total_before` | Сумарні втрати до впровадження DTLogix |
| Втрати після (грн/міс) | `loss_total_after` | Сумарні втрати після застосування `k_*` |
| Економія (грн/міс) | `savings_abs` | `Абсолютна економія = before − after` |
| Економія (%) | `savings_pct` | `Відносна економія = savings_abs / before` |
| Збережені години (год/міс) | `saved_hours` | Зекономлений час за місяць |

#### 2.2. Деталізація "До" (другий ряд / розкладка)

|  Поле  |  ID  |  Опис  |
| ------ | ---- | ------ |
| Час/місяць (год) | `time_total_per_month` | Сумарний час ручних |
| Вартість часу (грн/міс) | `cost_time_loss_per_month` | Час × ставка |
| Помилки даних (грн/міс) | `cost_errors_per_month` | Помилки × штраф |
| Демередж/простої (грн/міс) | `cost_demurrage_per_month` | Події × вартість |
| Відтік клієнтів (грн/міс) | `cost_churn_per_month` | К-сть клієнтів × CLV |

> **Примітка:** Ці 5 полів формують склад втрат "До" і відображаються як список із відсотком частки від `loss_total_before` або як stacked bar.

#### 2.3. Деталізація "Після" (опціонально)

|  Поле  |  ID  |  Опис  |
| ------ | ---- | ------ |
| Вартість часу після | `cost_time_loss_after` | З урахуванням `k_time` |
| Помилки після | `cost_errors_after` | З урахуванням `k_errors` |
| Демередж після | `cost_demurrage_after` | З урахуванням `k_demurrage` |
| Відтік після | `cost_churn_after` | З урахуванням `k_churn` |

### 3. Візуальні компоненти

#### 3.1. Підсвітка кольором KPI

- Якщо savings_pct ≥ 50% → бейдж "Сильний ефект".
- 30% ≤ savings_pct < 50% → "Відчутний ефект".
- 10% ≤ savings_pct < 30% → "Помітний ефект".
- < 10% → без бейджа (або "Є потенціал для покращення").

> **Примітка:** Колірна логіка (для дизайнера): зелені відтінки для позитивних трендів; нейтральні сірі для "до".

#### 3.2. Пояснювальний блок (текст)

1–2 речення, автогенеровані залежно від `savings_abs` і `saved_hours`.

**Шаблон**:

```txt
"Ви втрачаєте приблизно {loss_total_before} щомісяця. Завдяки DTLogix економія може скласти {savings_abs} ({savings_pct}), а це {saved_hours} год/міс, які повертаються в управління бізнесом."
```

### 4. Додаткові повідомлення та бейджі

- Якщо `loss_total_before` = 0 → інфоблок: "Здається, ви ввели нульові значення. Перевірте поля або скористайтеся рекомендованими прикладами."
- Якщо будь-який компонент > 50% від `loss_total_before` → бейдж "Головне джерело втрат" біля відповідного пункту (час / помилки / демередж / відтік).
- Якщо `savings_abs` ≥ 100 000 грн → бейдж "Високий ROI-потенціал".
- Якщо `saved_hours` ≥ 40 год/міс → бейдж "Мінус 1 робочий тиждень/міс" (для емоційної інтерпретації).

### 5. Приклади копірайту (динамічні)

- ROI-повідомлення:
  ```txt
  "DTLogix повертає вам до {savings_abs} на місяць та {saved_hours} годин операційного часу. Це прямий шлях до збільшення обороту без розширення штату."
  ```
- Фокус на найбільший канал втрат:
  ```txt
  "Найбільша частка втрат — {top_loss_label} ({top_loss_share}%). Саме тут DTLogix дає найсильніший ефект."
  ```
