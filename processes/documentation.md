# Стандарти документації

## Вступ

Стандарти документації забезпечують узгодженість у створенні PRD, user stories та прийнятті рішень. Документація має бути короткою, чіткою та прив'язаною до гіпотез та метрик.

## PRD (Product Requirements Document)

### Структура PRD

**1. Гіпотеза:**
- Що робимо, для кого, який очікуваний результат.
- Формат: "Ми вважаємо, що [функція] для [сегмент] призведе до [результат]."

**2. Проблема:**
- Яку проблему з Problems (Lean Canvas) закриваємо.
- Чому це важливо для користувача.

**3. Рішення:**
- Короткий опис функції (2–3 речення).
- Must-have vs nice-to-have частини.

**4. Метрики:**
- Які метрики вимірюємо.
- Baseline та очікуваний вплив.
- Критерій успіху.

**5. Пріоритет:**
- RICE score (якщо застосовується).
- Зв'язок з UVP та Problems.

**6. Acceptance criteria:**
- Критерії готовності функції.
- Як перевіряємо, що функція працює.

### Приклад PRD: Автоматичні сповіщення клієнтам

**Гіпотеза:**  
Ми вважаємо, що автоматичні сповіщення клієнтам про зміну статусу рейсу для сегменту A призведуть до зменшення вхідних дзвінків на 50% та підвищення adoption клієнтського лінку.

**Проблема:**  
Постійні дзвінки від клієнтів щодо статусу перевезення через відсутність прозорості (див. [Problems](/business-model/1-problems.md)). Експедитори витрачають 30–40% робочого часу на відповіді на повторювані запитання.

**Рішення:**  
Автоматична відправка сповіщень клієнтам при зміні статусу рейсу (loading → transit → customs → delivered) та при зміні ETA. Сповіщення відправляються через SMS та email. Must-have: сповіщення про зміну статусу. Nice-to-have: налаштування каналів сповіщень, шаблони повідомлень.

**Метрики:**  
- Baseline: середня кількість вхідних дзвінків від клієнтів (вимірюється через опитування та аналітику)
- Очікування: зменшення на 50% через 2 тижні після релізу
- Критерій успіху: зменшення на ≥ 50% + adoption клієнтського лінку ≥ 50% рейсів

**Пріоритет:**  
RICE = 80 (Reach = 100 користувачів/міс, Impact = 2, Confidence = 80%, Effort = 2 тижні). Зв'язано з UVP (економія часу) та Problems (зменшення дзвінків).

**Acceptance criteria:**  
- Коли статус рейсу змінюється, тоді сповіщення відправляється клієнту автоматично
- Сповіщення відправляються через SMS та email (доставка ≥ 95%)
- Кількість вхідних дзвінків від клієнтів зменшилася на ≥ 50% через 2 тижні
- Adoption клієнтського лінку ≥ 50% рейсів

## User Stories

### Формат user story

**Як [роль], я хочу [дія], щоб [ціль].**

**Приклад:**
- Як експедитор, я хочу автоматично відправляти сповіщення клієнтам при зміні статусу, щоб зменшити кількість дзвінків.

### Деталізація user story

**1. Опис:**
- Короткий опис (1–2 речення).

**2. Acceptance criteria:**
- Список критеріїв готовності (3–5 пунктів).
- Формат: "Коли [умова], тоді [результат]."

**3. Метрики:**
- Які метрики впливає story.
- Очікуваний вплив.

**4. Пріоритет:**
- Must-have або nice-to-have.
- Зв'язок з гіпотезою.

### Приклад user story

**Опис:** Як експедитор, я хочу автоматично відправляти сповіщення клієнтам при зміні статусу, щоб зменшити кількість дзвінків.

**Acceptance criteria:**
- Коли статус рейсу змінюється, тоді сповіщення відправляється клієнту автоматично.
- Коли сповіщення відправлено, тоді воно відображається в історії рейсу.
- Коли сповіщення не відправлено, тоді система логує помилку.

**Метрики:** Кількість вхідних дзвінків (очікування: -50%).

**Пріоритет:** Must-have для MVP.

## Формати прийняття рішень

### Рішення про пріоритети

**Формат:**
- Проблема / можливість.
- Варіанти рішень.
- Рекомендація з обґрунтуванням (RICE, зв'язок з метриками).
- Рішення.

**Приклад:**
- Проблема: Користувачі скаржаться на відсутність сповіщень.
- Варіанти: Автоматичні сповіщення vs ручні сповіщення.
- Рекомендація: Автоматичні сповіщення (RICE = 80, закриває Problem, впливає на метрику дзвінків).
- Рішення: Автоматичні сповіщення в наступному спринті.

### Рішення про корекцію

**Формат:**
- Гіпотеза та очікування.
- Фактичні результати (метрики).
- Аналіз розбіжностей.
- Варіанти дій: продовжувати, корегувати, відкотити.
- Рішення.

**Приклад:**
- Гіпотеза: Автоматичні сповіщення зменшать дзвінки на 50%.
- Результати: Дзвінки зменшилися на 30% (не досягли цілі).
- Аналіз: Сповіщення працюють, але не всі клієнти отримують (технічна проблема).
- Варіанти: Виправити технічну проблему vs додати SMS-канал.
- Рішення: Виправити технічну проблему, потім додати SMS-канал.

## Гіпотези

- Коротка, чітка документація прискорює розробку.
- Зв'язок з гіпотезами та метриками забезпечує фокус.
- Стандартизовані формати спрощують комунікацію.
- Acceptance criteria зменшують недоліки в розробці.

## Наступні дії / наслідки

- Кожна функція має PRD з гіпотезою та метриками.
- User stories мають чіткі acceptance criteria.
- Рішення документуються з обґрунтуванням.
- Документація оновлюється після валідації гіпотез.
