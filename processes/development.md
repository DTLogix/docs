# Процеси розробки

## Вступ

Процеси розробки забезпечують hypothesis-driven підхід, короткі цикли валідації та зв'язок між функціями та метриками з Lean Canvas.

## Принципи розробки

### Hypothesis-driven розробка

Кожна функція має чітку гіпотезу, яку валідуємо через метрики.

**Формат гіпотези:**
- Ми вважаємо, що [функція] для [сегмент користувачів] призведе до [результат].
- Ми знаємо, що успіх, якщо [метрика] зміниться на [значення] протягом [термін].

**Приклад:**
- Ми вважаємо, що автоматичні сповіщення клієнтам для сегменту A призведуть до зменшення дзвінків на 50%.
- Ми знаємо, що успіх, якщо кількість вхідних дзвінків зменшиться на ≥ 50% протягом 2 тижнів після релізу.

### Короткі цикли

**Ітеративна доставка:**
- Невеликі релізи (1–2 тижні) замість великих релізів.
- Швидка валідація гіпотез через метрики.
- Можливість корекції після кожного релізу.

**MVP-підхід:**
- Спочатку must-have функції, потім nice-to-have.
- Кожна функція має мінімальну версію для валідації.
- Розширення функцій на основі даних використання.

## Продуктова пріоритизація

### RICE framework

**Reach** — скільки користувачів зачепить функція за період (наприклад, місяць).

**Impact** — вплив на метрику (0.25 = мінімальний, 0.5 = низький, 1 = середній, 2 = високий, 3 = максимальний).

**Confidence** — впевненість в оцінках (50% = низька, 80% = середня, 100% = висока).

**Effort** — зусилля в людино-тижнях.

**Формула:** RICE Score = (Reach × Impact × Confidence) / Effort

**Приклад:**
- Автоматичні сповіщення: Reach = 100 користувачів/міс, Impact = 2 (високий), Confidence = 80%, Effort = 2 тижні
- RICE = (100 × 2 × 0.8) / 2 = 80

### Must-have / Nice-to-have

**Must-have для MVP:**
- Функції, без яких продукт не працює (створення рейсу, статуси, ETA).
- Функції, які критичні для валідації основної гіпотези (автоматичні сповіщення, клієнтський лінк).
- Пріоритет визначається через зв'язок з Problems та UVP.

**Nice-to-have:**
- Функції, які покращують досвід, але не критичні (AI-модуль, розширені сценарії документів).
- Можуть доїхати другим етапом без втрати цінності MVP.

**Критерій розподілу:**
- Must-have: без цієї функції користувач не отримає основну цінність (економія часу, зменшення хаосу).
- Nice-to-have: функція покращує досвід, але не критична для валідації гіпотези.

## Зв'язок з метриками

### Продуктові метрики

Кожна функція має прив'язку до метрик:

- **Автоматичні сповіщення** → кількість вхідних дзвінків, adoption клієнтського лінку
- **ETA** → точність ETA, churn rate
- **Клієнтський лінк** → adoption клієнтського лінку, кількість вхідних дзвінків
- **Централізоване управління** → час економії, MAU активних експедиторів

### Комерційні метрики

Функції впливають на комерційні метрики:

- **Економія часу** → retention, churn rate
- **Прозорість ETA** → лояльність клієнтів, churn rate
- **Простота онбордингу** → conversion demo → paid, CAC

### Вимірювання впливу

**Перед релізом:**
- Визначаємо baseline метрик.
- Формулюємо очікуваний вплив на метрики.

**Після релізу:**
- Вимірюємо зміни метрик через 1–2 тижні.
- Порівнюємо з baseline та очікуваннями.
- Приймаємо рішення: продовжувати, корегувати або відкотити.

## Гіпотези

- Hypothesis-driven підхід зменшує ризик розробки непотрібних функцій.
- Короткі цикли прискорюють валідацію та корекцію.
- Зв'язок з метриками забезпечує об'єктивність рішень.
- RICE framework допомагає пріоритизувати об'єктивно.

## Наступні дії / наслідки

- Кожна функція має гіпотезу перед початком розробки.
- Пріоритизація через RICE з прив'язкою до метрик.
- Вимірювання впливу після кожного релізу.
- Корекція пріоритетів на основі даних.
