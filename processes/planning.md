# Процеси планування

## Вступ

Процеси планування організують цикл від discovery до feedback, забезпечуючи швидкі ітерації та фокус на валідації гіпотез.

## Цикл розробки

### Discovery

**Мета:** Зрозуміти проблему та сформулювати гіпотезу.

**Дії:**
- Аналіз проблем з Lean Canvas (Problems, Customer Segments).
- Інтерв'ю з користувачами (якщо потрібно).
- Формулювання гіпотези: що робимо, для кого, який очікуваний результат.
- Визначення метрик для валідації.

**Результат:** Гіпотеза з метриками та очікуваннями.

### Design

**Мета:** Спроектувати рішення для валідації гіпотези.

**Дії:**
- Прототип або wireframe рішення.
- Визначення must-have vs nice-to-have функцій.
- Оцінка зусиль (Effort для RICE).
- Перевірка відповідності UVP та Problems.

**Результат:** Дизайн рішення з оцінкою зусиль.

### Development

**Мета:** Реалізувати мінімальну версію для валідації.

**Дії:**
- Розробка must-have функцій.
- Налаштування аналітики для відстеження метрик.
- Тестування на реальних сценаріях (якщо можливо).
- Підготовка до релізу.

**Результат:** Готова функція з аналітикою.

### Release

**Мета:** Випустити функцію та зібрати дані.

**Дії:**
- Реліз функції (див. [Релізи](./release.md)).
- Збір baseline метрик.
- Моніторинг використання та помилок.
- Збір фідбеку від користувачів.

**Результат:** Функція в продакшені з даними.

### Feedback

**Мета:** Валідувати гіпотезу та прийняти рішення.

**Дії:**
- Аналіз метрик через 1–2 тижні після релізу.
- Порівняння з baseline та очікуваннями.
- Збір фідбеку від користувачів.
- Прийняття рішення: продовжувати, корегувати, відкотити.

**Результат:** Рішення про подальші дії.

## Спринти

### Що виноситься у спринти

**Must-have функції:**
- Функції, критичні для валідації гіпотези.
- Функції з високим RICE score.
- Функції, які закривають Problems з Lean Canvas.

**Технічний борг:**
- Тільки якщо блокує розробку нових функцій.
- Пріоритет через вплив на швидкість розробки.

**Багфікси:**
- Критичні баги (блокують використання).
- Не критичні баги — в окремий backlog.

### Тривалість спринтів

**1–2 тижні:**
- Швидкі цикли для валідації гіпотез.
- Можливість швидкої корекції.
- Фокус на невеликих, але цінних функціях.

**Критерій:** Спринт має завершуватися релізом функції або частини функції.

## Iterative delivery

### Що вважається iterative delivery

**Мінімальна версія функції:**
- Must-have частина функції для валідації гіпотези.
- Nice-to-have частини можуть доїхати наступними ітераціями.

**Приклад:**
- Ітерація 1: Автоматичні сповіщення про зміну статусу (must-have).
- Ітерація 2: Сповіщення про зміну ETA (nice-to-have).
- Ітерація 3: Налаштування каналів сповіщень (nice-to-have).

**Критерій:** Кожна ітерація має самостійну цінність та може бути валідована через метрики.

### Переваги iterative delivery

- Швидка валідація гіпотез.
- Можливість корекції після кожної ітерації.
- Зменшення ризику розробки непотрібних функцій.
- Швидший feedback loop з користувачами.

## Зв'язок з метриками

### Планування метрик

**Перед початком розробки:**
- Визначаємо baseline метрик.
- Формулюємо очікуваний вплив на метрики.
- Налаштовуємо аналітику для відстеження.

**Під час розробки:**
- Перевіряємо, що аналітика налаштована.
- Готуємо дашборд для моніторингу метрик.

**Після релізу:**
- Вимірюємо зміни метрик.
- Порівнюємо з очікуваннями.
- Приймаємо рішення на основі даних.

## Приклад: User Story для DTLogix

### User Story: Автоматичні сповіщення клієнтам

**Опис:**  
Як експедитор, я хочу автоматично відправляти сповіщення клієнтам при зміні статусу рейсу, щоб зменшити кількість вхідних дзвінків та економити час.

**Acceptance criteria:**
- Коли статус рейсу змінюється (loading → transit → customs → delivered), тоді сповіщення відправляється клієнту автоматично
- Коли сповіщення відправлено, тоді воно відображається в історії рейсу з часом відправки
- Коли сповіщення не відправлено (помилка доставки), тоді система логує помилку та показує попередження експедитору
- Коли ETA змінюється більш ніж на 2 години, тоді клієнт отримує окреме сповіщення про зміну ETA
- Сповіщення відправляються через SMS та email (доставка ≥ 95%)

**Метрики:**
- Кількість вхідних дзвінків від клієнтів (очікування: -50% через 2 тижні)
- Adoption клієнтського лінку (очікування: ≥ 50% рейсів)
- Час, який економить експедитор (очікування: 30–40 хв/день на відповіді клієнтам)
- Доставка сповіщень (очікування: ≥ 95%)

**Пріоритет:**  
Must-have для MVP. Зв'язано з гіпотезою: автоматичні сповіщення зменшують дзвінки на 50%.

## Гіпотези

- Короткі цикли (1–2 тижні) прискорюють валідацію гіпотез.
- Iterative delivery зменшує ризик розробки непотрібних функцій.
- Зв'язок з метриками забезпечує об'єктивність рішень.
- Фокус на must-have спочатку прискорює досягнення цінності.

## Наступні дії / наслідки

- Кожен спринт має чітку гіпотезу та метрики для валідації.
- Спринти тривалістю 1–2 тижні з обов'язковим релізом.
- Iterative delivery для швидкої валідації та корекції.
- Регулярний аналіз метрик після кожного релізу.
